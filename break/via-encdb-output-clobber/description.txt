Bug title: read -o can overwrite enc.db and destroy other users' data
Bug type: integrity
Severity: 100 points

Command used:
  Register alice and bob, write a file for each, then read alice's file with
  -o enc.db. Attempt to read bob's file afterward.

Observed behavior:
  Alice's read -o enc.db succeeds and replaces the database with alice's file
  bytes. Bob's subsequent authenticated read fails with invalid/255.

Expected behavior:
  stor must reject output paths that would overwrite enc.db. Bob's data must
  remain intact.

Why this violates the spec or security goal:
  A user who only knows their own key can corrupt the entire encrypted database,
  including other users' files. This defeats per-user integrity protection.

Suspected file and line:
  stor108.c read handler: fopen(outfile, "wb") with no guard against enc.db.

Minimal reproducer:
  stor -u alice -k alicekey -f alicefile -o enc.db read
